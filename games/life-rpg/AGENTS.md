# LOVEISHOW 에이전트 운영 지침

## 범위와 우선순위
- 본 파일이 위치한 디렉터리 트리에 속한 모든 작업에 적용됩니다.
- 더 깊은 경로에 별도 `AGENTS.md`가 존재하면 해당 지침을 우선 적용합니다.
- 사용자·개발자 메시지의 직접 지시는 항상 최상위 우선순위입니다.

## 기본 언어·톤
- 소통 언어는 기본적으로 한국어를 사용합니다. 코드, 명령어, 식별자는 영문 그대로 유지합니다.
- 설명은 간결하고 친절하게, 결과 중심으로 작성합니다.

## 수정 전 승인 절차
1. **3줄 요약**: 무엇을, 왜, 어떻게 변경할지 한 줄씩 정리합니다.
2. **구체적 수정 계획**: 실행 가능한 단계(5–8개 내외)로 제시하고, 파일 경로·영향 범위·타입 변경 여부 등을 포함합니다.
3. **명시적 확인**: “위 계획으로 진행할까요? (예/아니오, 수정사항 제안 가능)” 형태로 사용자의 동의를 받은 뒤에만 변경을 수행합니다.

## 커뮤니케이션 및 도구 사용
- 무거운 터미널 명령 실행 전에는 1–2문장으로 사전 안내합니다.
- 텍스트/코드 검색은 `rg`를 우선 사용하며, 파일 편집은 `apply_patch`를 사용합니다.
- 큰 파일은 250줄 이하 단위로만 확인합니다.
- 빌드/린트/테스트 등 무거운 명령은 사용자 확인 후 실행합니다.

## 코드 변경 원칙
- 문제의 근본 원인을 최소 변경 범위로 해결하며 기존 스타일을 유지합니다.
- 관련 없는 리팩터링, 대규모 포맷팅, 패키지 추가를 지양합니다.
- TypeScript에서는 `null`과 `undefined`를 명확히 구분하고 선언부-반환 타입을 일치시킵니다.
- 테스트와 검증은 변경 영역에 맞춰 최소 범위에서 우선 수행합니다.

## Next.js / ESLint 가이드
- ESLint 경고는 기능에 영향이 있을 때만 수정합니다. 타입 오류는 반드시 해결합니다.
- `<img>`를 `next/image`로 바꾸는 작업은 성능·품질 이득이 명확할 때만 수행합니다.

## 보안 및 환경 변수
- 비밀 키를 코드에 하드코딩하지 않습니다. `.env.local` 등 환경 파일을 사용하고 Git에 커밋하지 않습니다.
- `OPENAI_API_KEY`, `GEMINI_API_KEY`, Firebase 등의 민감 키는 서버/클라이언트 사용 범위를 구분합니다.
- 로그에는 민감 정보를 그대로 출력하지 않고 앞 6–10자만 남기고 마스킹합니다.

## 스토리보드 모듈 특화 지침
- `/api/storyboard` 응답은 `{ ok: boolean }`을 기준으로 분기하며, 실패 시 `reason`을 제공합니다.
- 스타일 조회 실패 시 내장 Fallback을 우선 사용하고, 없으면 명확한 오류를 반환합니다.
- 자연어(`natural`)와 JSON(`json`) 모드를 분리하여 프롬프트를 구성합니다.

## 금지 사항
- 사용자 요청 없이 대규모 리네이밍, 포맷팅, 폴더 이동을 수행하지 않습니다.
- `rm -rf`, `git reset --hard` 등 파괴적 명령은 허가 없이 실행하지 않습니다.
- 외부 네트워크 접근, 설치, 권한 상승이 필요한 작업을 무통보로 진행하지 않습니다.

## 작업 종료 시 보고
- 변경 파일, 핵심 차이, 검증 결과를 간단히 요약합니다.
- 필요 시 다음 권장 작업을 1–2개 제안합니다.
